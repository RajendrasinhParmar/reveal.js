What is Version control?

Version control is a system that records changes to a file or set of files over time so that you can recall specific versions later

Version control software keeps track of every modification to the code in a special kind of database.

Goals Of Good Version Control System

  • Change History
    • A complete change history of every file
    • history should also include the author, date and written notes on the purpose of each change
    • This helps us identify and find the root cause of bug
  • Branching
    • Creating a branch keeps multiple stream of work independent from each other
    • This allow team members to collaborate so that different developer can work on different functionality at the same time
  • Merging
    • Ability to Merge two different branch one the work is done
    • It provide feature to properly merge changes without conflicts
  • Traceability
    • As we have a complete file history VCS enables us to trace all the changes made to the file
    • Some VCS also provides integration with different project management softwares like JIRA
    • This can be helpful in estimating future work with more accuracy
  • Comparison
    • Two version of same file
    • Two different branch
    • Two different tags
    • Any two different commits

Types of version control systems

  • Local Version Control Systems
  • Centralized Version Control Systems
  • Distributed Version Control Systems

Local Version Control Systems

  • Revision Control System (RCS) is an early version control system (VCS)
  • Patch sets (that is, the differences between files)
  • Allows users to make their own revisions of a document, commit and merge them together

Centralized Version Control Systems

  • The next major issue is to collaborate on different system. RCS doesn't provide it as it's local system
  • Centralized Version Control Systems (CVCSs)
  • CVS, Subversion(SVN), and Perforce are example of CVCS systems

Distributed Version Control Systems

  • Distributed Version Control Systems (DVCSs)
  • Git, Mercurial, Bazaar, Darcs
  • Client fully mirror the repository, including its full history
  • It allows us to have multiple remote repository to collaborate with different groups of people in different ways simultaneously within the same project

Git Commands

Add

This command will add file to staging area

$ git add FILE_NAME1, FILE_NAME2
$ git add .
This can also consider as marking file for commit 1 Will add two files to staging area 2 Will add all the file recursively to staging area from current dir

Log

This command will log the commit messages on current branch

$ git log

Author: Rajendrasinh Parmar 
Date:   Wed May 2 00:49:10 2018 +0530

	Add presentation for Git basics
can use different keys to get required commit elements

Commit

This command will commit the file(s) added to staging area

$ git commit -m "COMMIT_MESSAGE"
This will actually commit all the changes from staging area and record them in version control

Checkout

This command will checkout the branch

$ git checkout BRANCH_NAME
$ git checkout -b NEW_BRANCH_NAME
Create new branch using existing branch as base branch passing -b argument

Fetch

This command will fetch all the changes from remote repository to the local remote branch but will not merge them to actual local branch

$ git fetch REMOTE_NAME BRANCH_NAME
Fetch will only fetch changes to local remote branch that is tracking the actual remote branch

Merge

Merge all the changes from source branch the destination branch

$ git merge SOURCE_BRANCH
Fetch will only fetch changes to local remote branch that is tracking the actual remote branch

Pull

This command will pull all the changes from remote repository and merge them into local branch

$ git pull REMOTE_NAME BRANCH_NAME
Pull is combination of git fetch + git merge

PUSH

This command will push all the committed changes from local branch to remote branch

$ git push REMOTE_NAME BRANCH_NAME
Push will only push committed changes

REBASE

This is advanced git command that can be used to rewrite the git history

$ git pull REMOTE_NAME SOURCE_BRANCH
$ git checkout DESTINATION_BRANCH
$ git pull REMOTE_NAME DESTINATION_BRANCH
$ git rebase SOURCE_BRANCH
Make sure to have all the changes for both the branches locally
